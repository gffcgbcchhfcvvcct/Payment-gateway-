Integrating NexaPay Mobile Money API in Under 15 Minutes

A complete step-by-step guide to wiring up mobile money payments with the NexaPay REST API — from sandbox registration to processing your first live transaction. Covers Node.js, Python, and PHP with real code examples for all four countries.

What You Will Build

By the end of this guide, your application will be able to initiate a mobile money payment request, receive a real-time webhook when the payment completes, and display the result to your user — all in under 50 lines of code per language.

Step 1: Get Your API Keys

Sign up at nexapay.africa and navigate to your dashboard. Under Settings → API Keys, generate a new sandbox key pair. You will receive a sk_test_... secret key and a pk_test_... publishable key. Keep your secret key server-side only — never expose it in frontend code.

Step 2: Make Your First Charge (Node.js)

Install the SDK: npm install @nexapay/node

"The NexaPay SDK wraps our REST API with sensible defaults. Error handling, retries, and network-specific quirks are all managed for you — so you can focus on your product, not the plumbing." — NexaPay Engineering Team

Step 3: Receive Webhook Events

Configure your webhook URL in the dashboard under Settings → Webhooks. NexaPay will POST signed events to your endpoint when a payment completes, fails, or is reversed. Always verify the HMAC-SHA256 signature using your webhook secret before processing any event.

  • Listen for payment.completed — trigger fulfillment
  • Listen for payment.failed — prompt retry or alternate method
  • Listen for refund.completed — notify the customer

Step 4: Go Live

Replace your sandbox keys with production keys. Update your webhook URL. Submit your merchant verification documents if not already done. Once approved, you are live across all four countries immediately.

Webhook Best Practices: Building a Reliable Payment Event System

Webhooks are the backbone of a responsive payment integration. When a mobile money payment is confirmed, your system needs to know immediately — not on the next polling cycle. Here is everything you need to build a bulletproof webhook handler for NexaPay.

1. Always Verify the Signature

Every NexaPay webhook POST includes an X-NexaPay-Signature header. This is an HMAC-SHA256 hash of the raw request body using your webhook secret. Never process a webhook event without verifying this signature — it prevents replay attacks and spoofed events.

2. Return 200 Immediately, Process Asynchronously

Your webhook endpoint should respond with HTTP 200 within 5 seconds. If your handler takes longer, push the event to a queue (Redis, RabbitMQ, SQS) and process it in the background. NexaPay will retry events that receive non-2xx responses or timeout, up to 5 times with exponential backoff.

3. Handle Duplicate Events Idempotently

Due to network retries, you may occasionally receive the same event more than once. Use the event's unique event_id to check whether you have already processed it before taking action.

  • Store processed event_id values in a Redis set or database table
  • On receipt, check if the ID exists before processing
  • Set a TTL of 7 days on stored event IDs

4. Use a Dedicated Webhook Endpoint

Create a dedicated /webhooks/nexapay endpoint with signature verification middleware, raw body parsing, and structured logging for every event received.

5. Monitor Your Webhook Health

Use the NexaPay dashboard's Event Inspector to see every event sent to your endpoint, its delivery status, and retry history. Set up alerts for sustained failure rates above 5%.

"A webhook that silently fails is worse than no webhook at all. Build monitoring and alerting from day one, not as an afterthought." — NexaPay Developer Experience Team

Accepting EcoCash Payments in Zimbabwe: A Developer's Deep Dive

EcoCash is the dominant mobile money network in Zimbabwe, holding over 70% market share with more than 8 million active wallets. For any business operating in Zimbabwe, accepting EcoCash is non-negotiable. Here is how to integrate it via NexaPay with minimal friction.

Zimbabwe's Unique Currency Environment

Zimbabwe operates a dual-currency environment — the Zimbabwe Gold (ZWG) and USD. EcoCash supports both, though USD wallets have higher limits and are preferred for larger transactions. NexaPay automatically routes USD charges to USD-capable wallets and ZWG charges to local wallets.

PIN Push vs. USSD Flow

EcoCash in Zimbabwe uses a PIN push model — your customer receives an on-screen prompt on their phone and enters their EcoCash PIN to authorise the payment. The NexaPay checkout flow handles this natively — you simply initiate the charge and we manage the customer communication.

Handling Zimbabwe-Specific Edge Cases

  • Dual currency routing: Specify currency: "USD" for dollar transactions and currency: "ZWG" for local currency.
  • Network downtime: EcoCash occasionally experiences scheduled maintenance windows. NexaPay automatically queues and retries charges during these windows.
  • Transaction limits: EcoCash USD wallets have a $1,000 per-transaction limit and $5,000 per-day limit.
  • Telecash and OneMoney: Both networks are supported via the same API endpoint — just change the network parameter.

Testing in Sandbox

Our sandbox includes test phone numbers that simulate different EcoCash scenarios: successful payments, insufficient funds, PIN timeout, and network timeout.

Integrating NexaPay with Python: Django & Flask Step-by-Step

Python remains one of the most popular server-side languages among African developers. Whether you're building with Django, Flask, or FastAPI, the NexaPay Python SDK makes wiring up mobile money payments clean and straightforward.

Installing the SDK

Install via pip: pip install nexapay. Then set your secret key as an environment variable: NEXAPAY_SECRET_KEY=sk_live_.... Never hard-code keys in source files.

Initiating a Charge in Django

In your Django view, import the client and call nexapay.charges.create() with the customer's phone number, amount, currency, and country code. The SDK returns a charge object with a status field and a charge_id you should store for reconciliation.

Flask Webhook Handler

Create a dedicated route at /webhooks/nexapay. Use Flask's request.get_data() to get the raw body — never use request.json before signature verification, as JSON parsing may alter byte ordering. Verify the HMAC-SHA256 signature using the raw body and your webhook secret.

  • Use hmac.compare_digest() for constant-time signature comparison
  • Return 200 OK immediately and process asynchronously with Celery
  • Log every event to your database before processing
"The Python SDK handles retry logic, timeout management, and network-specific formatting for all four countries. You just handle your business logic." — NexaPay SDK Team

NexaPay for PHP: Laravel Integration & WooCommerce Plugin Guide

PHP powers a huge proportion of African e-commerce — from bespoke Laravel applications to WordPress/WooCommerce shops. This guide covers both: a clean Laravel service class implementation, and setting up the official NexaPay WooCommerce plugin.

Laravel Service Class

Install the SDK via Composer: composer require nexapay/nexapay-php. Create a NexaPayService class in app/Services/ that wraps the SDK client. Inject it via Laravel's service container and bind it as a singleton in your AppServiceProvider.

Handling the Webhook in Laravel

Register a route in routes/api.php that is excluded from CSRF middleware. Use the raw request body (via Request::getContent()) for signature verification before calling json_decode(). Use queued jobs to process events asynchronously.

WooCommerce Plugin

The NexaPay WooCommerce plugin adds a "Pay with Mobile Money" gateway to your checkout. After installing the plugin, enter your API keys in WooCommerce → Settings → Payments → NexaPay. The plugin automatically handles order status updates via webhook — no additional configuration is required.

  • Compatible with WooCommerce 7.x and 8.x
  • Supports all four countries and networks out of the box
  • Test mode toggle uses your sandbox keys for safe testing

PCI DSS for African Merchants: What You Need to Know and What NexaPay Handles for You

PCI DSS compliance sounds daunting — 12 requirements, annual audits, and technical jargon that can overwhelm even experienced developers. But for most merchants on NexaPay, your compliance burden is dramatically reduced. Here is exactly how it works.

The PCI DSS Levels Explained

PCI DSS has four merchant levels based on annual transaction volume. Level 1 (>6 million transactions) requires a full annual on-site audit by a QSA. Level 4 (<20,000 e-commerce transactions) only requires a self-assessment questionnaire (SAQ). Most NexaPay merchants qualify as Level 3 or Level 4.

What NexaPay's Level 1 Certification Means for You

Because NexaPay is PCI DSS Level 1 certified and all payment data is processed entirely within our infrastructure, most merchants integrating via our API qualify for SAQ-A — the simplest possible self-assessment. This is a 22-question form, not a full audit.

Your Responsibilities

  • Never log or store card numbers, CVVs, or PINs in your own systems
  • Use HTTPS (TLS 1.2+) on all pages of your site
  • Keep your server software and dependencies updated
  • Restrict admin access using strong passwords and MFA
  • Complete the annual SAQ-A self-assessment honestly
"By using NexaPay as your payment processor, you've already addressed the hardest parts of PCI compliance. What remains for most merchants is common-sense security hygiene." — NexaPay Compliance Team

Building Bulletproof Payment Flows: Idempotency, Retries & Graceful Error Handling

Networks fail. Servers restart. Mobile money operators have maintenance windows. A payment integration that only works in perfect conditions will fail your customers in the real world. Here is how to build for resilience from the start.

Always Use Idempotency Keys

Every charge request to NexaPay should include a unique Idempotency-Key header. This is typically a UUID v4 generated per charge attempt. If you retry a request with the same key, NexaPay returns the result of the first attempt — no duplicate charge is created. Generate the key before the request and store it alongside your pending order record.

Implement Exponential Backoff

When a charge request returns a 5xx error or times out, do not retry immediately. Wait 1 second, then 2 seconds, then 4, then 8 — up to a maximum of 32 seconds. After 5 attempts, surface the failure to your user and suggest they try again or use an alternative payment method.

Handle Every Status, Not Just Success

  • payment.completed — fulfil the order
  • payment.failed — prompt retry, log the failure reason
  • payment.pending — show a "waiting for confirmation" state; set a 5-minute timeout
  • payment.reversed — roll back any fulfillment, notify the customer

Test Failure Scenarios in Sandbox

Our sandbox provides test phone numbers that simulate every failure mode: insufficient funds, wrong PIN, SIM swap block, network timeout, and operator maintenance window. Test all of them before going live.

"The difference between a 95% and a 99.5% success rate on payment flows is almost entirely determined by how well you handle edge cases. Sandbox testing edge cases is non-negotiable." — NexaPay Engineering Team
