Security is not a feature at NexaPay — it is a foundational requirement. Every architectural decision, every dependency choice, every deployment process is evaluated through a security lens. This page describes in technical detail how we protect your payments, your customers' data, and your business.
01
Security Overview

NexaPay processes real-money mobile money transactions on behalf of merchants across four Southern African countries. The stakes are high: every security failure has direct financial consequences for real people. Our security programme is designed to prevent, detect, and respond to threats at every level.

Zero Trust Architecture
Every service-to-service call is authenticated. No implicit trust between internal components.
Secrets Management
No plaintext credentials stored anywhere. All secrets managed via encrypted vaults with strict access controls.
24/7 Monitoring
Continuous anomaly detection with automated alerting for unusual transaction patterns, API abuse, and infrastructure events.
Regulatory Compliance
Aligned with POPIA (SA), PDPA (Zambia), and data protection regulations in each operating country.
02
Transport Security

All data transmitted between your application and NexaPay, between NexaPay and mobile money operators, and between our internal services, is encrypted using TLS 1.3. We do not accept connections over TLS 1.0 or TLS 1.1.

  • TLS 1.3 enforced — Minimum TLS 1.2 with TLS 1.3 preferred. Older protocol versions are rejected at the load balancer layer.
  • Strong cipher suites — We use ECDHE key exchange with AES-256-GCM and ChaCha20-Poly1305. Weak ciphers (RC4, 3DES, export-grade) are explicitly disabled.
  • HSTS — HTTP Strict Transport Security with a max-age of 1 year, including subdomains, is set on all API and dashboard endpoints.
  • Certificate pinning — Our mobile SDK optionally supports certificate pinning for high-security deployments.
  • HTTPS everywhere — All HTTP requests are permanently redirected to HTTPS. There are no unencrypted API endpoints.

Our API endpoints receive an A+ rating on Qualys SSL Labs. We proactively monitor for new TLS vulnerabilities and patch within 24 hours of public disclosure.

03
Data Encryption at Rest

All sensitive data stored by NexaPay — including API keys, merchant credentials, KYC documents, transaction records, and personal information — is encrypted at rest using AES-256 symmetric encryption.

  • Database encryption — All database volumes are encrypted at rest. Database credentials are never stored in application code or environment files.
  • API key storage — API keys are stored as salted bcrypt hashes. We never store the plaintext value. If an API key is lost, it must be regenerated — we cannot recover it.
  • KYC document storage — Identity documents uploaded for KYC are stored in an encrypted object store with access controlled by short-lived signed URLs.
  • Field-level encryption — Particularly sensitive fields (e.g. phone numbers, bank account details) use additional field-level encryption on top of full-volume encryption.
  • Key rotation — Encryption keys are rotated on a 90-day schedule. All historical data is automatically re-encrypted during rotation.
04
Authentication & Access Control

NexaPay uses multiple layers of authentication to protect merchant accounts, API access, and internal systems.

  • API key authentication — All API calls must include a valid secret key in the Authorization header. Keys have configurable scopes (read-only, full access) and can be restricted by IP range.
  • Dashboard 2FA — Two-factor authentication via TOTP (Google Authenticator, Authy) is available for all merchant dashboard accounts and strongly recommended.
  • 3DS payment authentication — NexaPay supports 3D Secure authentication for card-not-present transactions, adding an additional cardholder verification step.
  • Session management — Dashboard sessions expire after 30 minutes of inactivity. Sessions are invalidated server-side on logout — not just client-side cookie deletion.
  • Rate limiting — All API endpoints are rate-limited per API key. Brute-force attempts trigger automatic account protection and notification.
  • Internal access — Internal systems use role-based access control (RBAC) with the principle of least privilege. Production database access requires multi-factor authentication and is logged and audited.

API keys for production and sandbox environments are completely separate. Sandbox keys can never be used against the live payment network and vice versa.

05
Fraud Detection & Prevention

NexaPay operates a multi-layer fraud detection system that analyses every transaction in real time before it is submitted to the mobile money network.

  • Velocity checks — Automated rules flag transactions that exceed configurable velocity thresholds (e.g. more than N transactions per phone number per hour).
  • Geolocation anomaly detection — Transactions originating from unexpected geographic locations are flagged for review.
  • Duplicate detection — Idempotency key enforcement prevents accidental double-charges. Our system also detects non-idempotent duplicates within a configurable window.
  • Blacklists — Known fraudulent phone numbers, IP addresses, and merchant IDs are maintained in a real-time blocklist.
  • Merchant-configurable rules — Merchants can set their own transaction limits, block certain phone number patterns, and require additional verification for high-value transactions.
  • Chargeback prevention — Our webhook system ensures merchants receive immediate notification of payment status, reducing disputed transactions.
06
Data Residency & Sovereignty

NexaPay takes data sovereignty seriously. We understand that payment data involving Southern African users and merchants has regulatory implications for where that data may be stored and processed.

  • Primary data residency — Merchant and transaction data is stored primarily in data centres located within Southern Africa.
  • Cross-border transfer controls — Any transfer of personal data outside the originating country complies with the applicable data protection law, including POPIA (South Africa), the Data Protection Act (Zambia), and equivalent legislation.
  • Data minimisation — We collect and retain only the data necessary for payment processing, compliance, and fraud prevention. Unnecessary data is purged on schedule.
  • Retention periods — Transaction records are retained for 7 years for tax and regulatory compliance. KYC documents are retained as required by applicable anti-money laundering regulations.
07
Infrastructure Security

Our infrastructure is hardened against both external and internal threats using industry-standard practices.

  • Network segmentation — Payment processing services, customer data stores, and management systems operate in separate network segments with strictly controlled ingress/egress rules.
  • Immutable infrastructure — Production servers are never modified in place. All deployments use immutable images built from a verified supply chain.
  • Dependency management — All third-party dependencies are pinned to specific versions and scanned for known CVEs on every build. Critical vulnerabilities block deployment.
  • Secrets rotation — All infrastructure credentials (database passwords, service tokens) are rotated automatically on a regular schedule.
  • Backup & recovery — Database backups run every 6 hours. Point-in-time recovery is available for all production databases. Backup restoration is tested monthly.
  • DDoS protection — All API endpoints are protected by Layer 3/4 and Layer 7 DDoS mitigation.
08
Penetration Testing

NexaPay conducts regular security assessments to proactively identify and remediate vulnerabilities before they can be exploited.

  • Annual external penetration tests — Conducted by independent security firms with expertise in financial services and API security.
  • Continuous automated scanning — SAST (Static Application Security Testing) runs on every code commit. DAST (Dynamic Application Security Testing) runs weekly against our staging environment.
  • OWASP Top 10 coverage — All API endpoints and dashboard interfaces are regularly tested against the OWASP Top 10 web application security risks.
  • Remediation SLAs — Critical findings are remediated within 24 hours. High findings within 7 days. Medium findings within 30 days.

Penetration test results and full compliance attestation reports are available to enterprise merchants and regulated institutional customers upon request under NDA.

09
Responsible Disclosure

We welcome reports from security researchers who discover vulnerabilities in NexaPay's platform. We are committed to working with the security community to keep our platform safe.

To report a security vulnerability:

  • Email security@nexapay.com with a detailed description of the vulnerability.
  • Include steps to reproduce, the potential impact, and any proof-of-concept code (no live exploitation required).
  • We will acknowledge your report within 24 hours and provide an initial assessment within 72 hours.
  • We ask that you do not publicly disclose the vulnerability until we have had a reasonable opportunity to investigate and remediate it (typically 90 days).
  • We do not take legal action against researchers who act in good faith and follow these guidelines.

We thank all researchers who help keep NexaPay secure. If your report leads to a valid vulnerability fix, we will acknowledge your contribution publicly (with your permission).
